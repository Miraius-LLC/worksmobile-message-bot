# 🏆 第 n 回 エージェント対抗 `<repo名>` 大レビュー大会 (3 エージェント三角測量レビュー)

大規模なコードベースを、**claude / codex / agy の 3 エージェントが全員・全観点を独立にレビュー**し、
結果を 1 本の文書に統合する「レビュー大会」。狙いは「1 人の見落とし」を 3 者の独立判断で潰すこと。
最後に **殊勲賞 (MVP) 表彰式** で、他者が気づけなかった貴重な単独指摘を讃える。

## 大会名と「第 n 回」の決め方

- 正式名称: **`第 n 回 エージェント対抗 <repo名> 大レビュー大会`** (例:「第 3 回 エージェント対抗 501 大レビュー大会」)。
- `<repo名>` = 対象 repo の basename (`~/Develop/<project>` の basename。`~/Develop` 自体は `develop-meta`)。
- **n は「その repo の過去開催数 + 1」**。`docs/reviews/code-review-*.md` の本数を数えて決める
  (`ls docs/reviews/code-review-*.md 2>/dev/null | wc -l` → +1)。0 本なら第 1 回。
- 依頼文・サマリー・表彰式の発表文で、この大会名をキャッチーに使う (士気が上がるし、後から追える)。

## この skill の中核原則 (絶対に崩さない)

1. **3 者は対等・全員が全観点をやる。** 観点を専門で分担しない。File 用 / Security 用…と役割分割しない。
   claude も codex も agy も、下の 7 観点すべてを同一スコープに対して通しでレビューする。重み付けしない。
2. **独立・blind・カンニング厳禁 (大会の生命線)。** 各エージェントは**他者の指摘を一切見ずに**自分だけで判断する。
   - 並列にディスパッチし、ある agent の出力を別 agent のプロンプトに入れない。
   - 「他のエージェントはこう言っている」「先行レビューでは…」を依頼文に**絶対書かない**(anchoring で独立性が壊れる)。
   - **まとめ役 (主エージェント) も出場選手の 1 人。自分の独立票を MD に書き出して保存・確定するまで、
     他 2 者の返信 MD を一切開かない** (詳細は Phase 1 の「カンニング防止プロトコル」)。これを破った票は**汚染票**として
     無効化し、別の未汚染 agent に投げ直す。成果物にも「○○ 票は手順違反で除外」と明記する (隠さない)。
3. **同調しない。** 各 agent は忖度・気兼ねで相手に合わせない。技術的根拠だけで判断する (§10 3 原則 #2)。
   統合段階でも相違を均さない。3 者が**独立に**到達した一致だけを「高信頼」とし、割れたものは「意見差分」で残す。
4. **レビュー判断は必ず agent-room 経由 (確定)。** ~/Develop/AGENTS.md §10 の絶対ルール。自セッション内の
   内部 self-review で他者レビューを代替しない。実レビューは agent-room delegate で実 agent に投げて取得する。
5. **完全 read-only (絶対)。** レビュアーはコードを書き換えない・worktree を作らない・実装しない・commit しない。
   見るだけ。各自の指摘は **MD ファイルに保存して報告**し、**まとめ役 (主エージェント) がそれらを 1 本に統合**する。

## レビュー 7 観点 (全 agent が全部やる)

1. **ファイル単位のコードレビュー** — 可読性 / 責務分離 / 命名 / 重複 / 設計の違和感
2. **共通化・保守性** — 共通化すべき処理 / 抽象化すべき箇所 / 逆に共通化しすぎない方が良い箇所 / 将来変更に弱い構造
3. **パフォーマンス** — 無駄な処理 / N+1 や過剰なループ / DB・API 呼び出しの非効率 / キャッシュ・遅延評価・バッチ化の余地
4. **テスト不足** — 単体 / 結合が要る箇所 / 境界値・異常系・権限系の不足 / 回帰テスト化すべき重要処理
5. **バグ・エラー温床** (`diagnose` skill 観点) — 例外処理不足 / null・undefined 想定漏れ / 型の過信 / 状態不整合 /
   日付・権限・トランザクション周りの危険箇所 / 本番障害につながる箇所
6. **セキュリティ・権限** — 認証・認可の弱点 / 入力検証不足 / 情報漏洩 / SQL・GraphQL・API 周りの脆弱性 /
   ログに出してはいけない情報 / 権限昇格の可能性。
   **DB がある場合は必ず**: RLS / 行レベル権限 / テナント分離 (cross-tenant 漏洩) / ロール境界
   (DB・外部入力由来の値を union/enum に `as` で無検証キャストして権限すり抜け、L11) / policy 依存列の変更漏れ (L24/L30)。
7. **不要コード・不要テストの削除提案** — デッドコード / 未使用 export・関数・依存 / コメントアウト放置 /
   形式だけで実質何も検証していないテスト / 仕様更新に追従できず古くなった (stale) テスト・doc・コメント。
   **削除「提案」に留める** (この skill では実削除しない)。

## 役割 (出場選手とまとめ役)

- **まとめ役 (オーケストレータ 兼 出場選手)** = この skill を起動した agent。スコープ確定・依頼書配布・統合・
  文書化・表彰式進行を担う。**同時に自分も独立票を 1 本投じる出場選手**であり、カンニング防止プロトコル (Phase 1) を守る。
  - **まとめ役・統合は Codex 寄りを既定候補にする**。Claude に固定担当領域は置かず、スコープが大きい場合や速度が重要な場合は、
    オーケストレーション自体を `agent-room delegate --to codex` でまとめ役ごと委譲する判断をしてよい (要所は藤井に一言)。
- **3 人の独立レビュアー (出場選手)** = claude / codex / agy。まとめ役は自分自身の独立レビューを 1 本担い、
  残り 2 agent に agent-room で同一依頼を投げる (自己トリガ禁止: claude は `--to claude` しない、codex は `--to codex`、
  agy は `--to agy` しない。自分の分は自セッションで実施する)。結果 3 者の独立レビューが揃う。

## 手順

### Phase 0 — 開催準備とスコープ確定 (まとめ役)

1. `git log --oneline -20` で HEAD を把握 (git-log ルール)。
2. **大会名を確定**: `<repo名>` を求め、`docs/reviews/code-review-*.md` の本数 + 1 で第 n 回を決める。
3. **既定スコープ＝リポジトリ全体**。引数でパス / commit range / PR を渡されたらそれに絞る。
   スコープが巨大で曖昧なら藤井に確認 (L2: ユーザー依存の判断は推測せず確認)。
4. 関連 doc を読む (L3): `CONTEXT.md` (用語集) / `AGENTS.md` / `CLAUDE.md` / `docs/adr/`。ドメイン語彙と決定を踏まえる。
5. `git ls-files` でソースを列挙 → ディレクトリ / モジュール単位の**レビュー単位 (chunk)** に分割。
   巨大 repo は chunk 化し、**各 chunk を 3 者全員がレビュー**する (chunk を agent 間で分担しない＝三角測量を保つ)。
6. **agent-room の健全性を確認**: `agent-room status --health` / `agent-room doctor`。
   codex / agy の poller が落ちていたら**黙って省略せず**藤井に報告して指示を仰ぐ (no silent caps)。

### Phase 1 — 3 者の独立レビュー (並列・blind)

#### 1-a. 依頼書を MD で `/tmp/` に書き出して配布する

依頼文を post 本文に長文で埋めず、**1 本の指示書 MD を `/tmp/` に書き出して配布**する (公平・独立・トークン節約)。

- 保存先: **`/tmp/<repo名>-review-<YYYY-MM-DD-HHMM>.md`** (JST。`Date.now()+9h` → `toISOString()`、L25)。
- 中身は下の「依頼書テンプレート」。**全 agent に同一の 1 ファイル**を読ませる (文面差で独立性を歪めない)。
- 依頼書に他者の見解を含めない。`repo:<project>`・大会名・対象 range/パス・7 観点・指摘フォーマット・
  **read-only**・**指摘を MD に保存して報告**することを明記。
- post 本文はごく短く、この指示書ファイルを読ませるだけにする (例:「指示書 `/tmp/...md` を読んでレビューしてください」)。

#### 1-b. 並列にディスパッチ (直列にしない＝後発が先発を見て同調するのを防ぐ)

- `agent-room delegate --from <self> --to codex "@codex repo:<project> 第 n 回 <repo名> 大レビュー大会の独立レビューです。指示書 /tmp/<repo名>-review-<...>.md を読んで単独でレビューしてください"`
- `agent-room delegate --from <self> --to agy   "@agy   repo:<project> 第 n 回 <repo名> 大レビュー大会の独立レビューです。指示書 /tmp/<repo名>-review-<...>.md を読んで単独でレビューしてください"`

#### 1-c. 🚨 カンニング防止プロトコル (まとめ役=出場選手の独立性担保)

**まとめ役自身も独立票を投じる出場選手である。** スクショの汚染事故 (他者の票を読んでから自票を書く) を構造で防ぐため、
以下の順序を**絶対**に守る:

1. 他 2 者を 1-b でディスパッチしたら、**すぐに自分の独立レビューに着手する**。
2. 自分の独立票を **MD ファイルに書き切って保存・確定する**:
   `.agent-room/<project>/reviews/<self>-<YYYY-MM-DD-HHMM>.md` (まとめ役自身の票もここに残す)。
3. **この自票の保存が完了するまで、他 2 者の返信 MD を一切開かない。**
   - `Monitor` / `agent-room watch` で **done の発生 (状態遷移) を観測するのは OK**。
   - だが **reviews/ の他者 md を Read しない・`agent-room reply` で本文を読まない**。中身を見たら自票は汚染票。
4. 自票を保存・確定してから初めて、Phase 2 で 3 本の MD を突き合わせる。
5. **万一カンニングした場合** (順序違反): その票は**汚染票として無効化**し、Claude/agy 既存票は保持、
   汚染した本人の票だけ無効扱いにして**別の未汚染 agent に同一スコープで投げ直す**。成果物に
   「○○ 票は手順違反で除外、△△ に再依頼」と**必ず明記**する (隠さない。スクショの運用ミスを繰り返さない)。

#### 1-d. 観測と受け取り

- **観測は速い経路で**: terminal セッションからは `post --watch` を background で待たず、
  `Monitor` で `agent-room watch --since <run> --until codex,agy` を回して done を速く拾う
  (~/Develop/AGENTS.md §10、commit 476d2ba)。長文全文は `agent-room reply <run_id>` か `reviews/` の md を Read
  (ただし **自票を確定する前は他者本文を開かない**、1-c)。
- **受け取り方** (`receiving-code-review` の規律): 返ってきた指摘を鵜呑みにも全否定もしない。
  技術的に妥当か file:line で裏取りする。ただし**この段階では自分の独立票を書き換えない** (Phase 2 で突き合わせる)。
- **L1 (confabulation 回避)**: 各レビューに `file_path:line` が伴うか検証。具体パスが無く一般論だけの返答は
  信頼せず、対象を絞って再依頼するか、まとめで「根拠薄」と明示する。

### Phase 2 — 統合 (まとめ役、1 本に集約)

3 者の独立レビュー (まとめ役の確定済み自票を含む) が揃ったら、**1 本のレビュー文書**に統合する。

- **まず重複排除 (合意度カウントの前提)**: 各指摘を **`file_path:行 + 種別`** で正規化キーにし、3 者の指摘を突き合わせる。
  同じ箇所・同じ種別を指していれば「同一指摘を n 者が独立に挙げた」と数える (行が数行ずれても、同じ問題を指していれば同一とみなす)。
  これをやらないと「3 者が同じ穴を見つけた」のか「別々の 3 件」なのか区別できず、合意度が無意味になる。
- **誰が挙げたかを必ず明示**: すべての指摘に **`by @claude` / `by @codex` / `by @agy`** を付し、合意度の内訳
  (例: `合意度 2/3 (by @claude, @codex)`) を書く。単独指摘も `合意度 1/3 (by @agy)` と発見者を明記する
  (表彰式の根拠になるし、後から「誰の眼が鋭いか」を追える)。
- **一致 (consensus)**: 3 者が**独立に**同じ指摘 → 高信頼。`合意度 3/3 (by @claude, @codex, @agy)`。
- **部分一致**: 2/3 → 中。1/3 → 要検討として残す (独自の鋭い指摘も握り潰さない)。
- **意見差分**: 評価が割れた箇所 → 丸めず「意見差分」として両論を併記する (どちらが優れているとは断じない)。
- **重大度順** (Critical → High → Medium → Low) に並べる。
- **推測と事実を分ける**。仕様が不明な箇所は「仕様確認事項」として残す (勝手に断定しない)。
- まとめ役は各 agent の意見に**優劣を付けない・独断で省略しない・勝手に丸めない**。

### Phase 3 — 🏆 殊勲賞 (MVP) 表彰式 (agent-room で開催)

統合が終わったら、**殊勲賞 (MVP) 表彰式**を agent-room で開催する。他 agent が気づけず単独で挙がった指摘
(合意度 1/3) を「貴重さ = 重大度 (Critical 重視) × 独自性」でランク付けし、**殊勲賞トップ 5** を発表する。

- 発表は **`agent-room delegate --from <self> --to codex,agy "..."`** で**他 agent を起こして**トップ 5 を共有し、
  それぞれの**感想**をもらう (例: この発見をどう見るか / 自分が見落とした理由 / 異論)。まとめ役自身の感想も添える。
  (自己トリガ禁止: まとめ役が claude なら `--to codex,agy`、codex なら `--to claude,agy`、agy なら `--to claude,codex`。)
- 発表文は大会名を冠してキャッチーに (例:「🏆 第 n 回 <repo名> 大レビュー大会・殊勲賞発表! 第 1 位は **by @agy** の…」)。
  各賞に**受賞 agent (`by @<agent>`)・file:line・なぜ貴重か**を明記する。
- 文書 (`docs/reviews/`) にトップ 5 と各 agent の感想を残す。

## 注意事項 (ガードレール)

- **完全 read-only (絶対)**。コードの書き換え・worktree 作成・実装・commit を一切しない。レビューと診断だけに集中する (修正は別タスク)。
- **カンニング防止 (Phase 1-c)**: まとめ役も出場選手。自票を MD に確定する前に他者票を開いたら汚染票 → 無効化・再依頼・成果物に明記。
- **agent-room 規律**: 自己トリガ禁止 / 依頼本文は日本語 (コード片・識別子・パス・range は原文可) /
  `repo:<project>` 明示 / read-only 明記 / deploy・本番操作はしない。
- 思想論に広げすぎない。具体のコード (file:line) に紐づけて語る。
- 重大度の高いものから優先的に整理する。
- 機微情報 (マイナンバー / 健康情報 / 口座番号) を成果物に出さない (~/.agents/AGENTS.md §4.4)。
- 成果物 `docs/reviews/` への commit は worktree 規律に従い、組織 repo (501 / asunaro 等) では push 前に藤井確認。

## 依頼書テンプレート (`/tmp/<repo名>-review-<...>.md` に書き出し、全 agent に同一文面)

```text
# 🏆 第 n 回 エージェント対抗 <repo名> 大レビュー大会 — 独立レビュー依頼書

repo:<project>
目的: 大規模コードレビュー (三角測量の独立 1 票)。あなたは他レビュアーの意見を見ずに、単独で全観点をレビューしてください。
対象: <commit range / PR / パス。無指定なら repo 全体 (HEAD)>
観点: 以下 7 つを全て見てください。
  1 ファイル単位 (可読性/責務/命名/重複/設計)
  2 共通化・保守性 (共通化/抽象化/過共通化回避/将来変更耐性)
  3 パフォーマンス (無駄処理/N+1/DB・API 非効率/キャッシュ余地)
  4 テスト不足 (単体/結合/境界・異常・権限/回帰)
  5 バグ・エラー温床 (例外/null/型過信/状態不整合/日付・権限・Tx)
  6 セキュリティ・権限 (認証認可/入力検証/情報漏洩/インジェクション/権限昇格。DB あれば RLS・行レベル権限・テナント分離・ロール境界も必須)
  7 不要コード・不要テストの削除提案 (デッドコード/未使用 export/形式だけ・stale なテスト/古いコメント・doc)
出力: 各指摘に file_path:line・重大度 (Critical/High/Medium/Low)・根拠・改善案・テスト観点。推測と事実を分ける。
      指摘は MD にまとめて報告 (reviews/ に全文)。**MD 末尾に自分の完走時間・使用トークン概算を必ず記す** (コスパ評価用)。

```
